Product Requirements Document: Automated Security Evaluation Pipeline for Agent Skills1. Problem Statement and Strategic ContextThe paradigm of agentic software development has experienced a fundamental shift with the introduction and widespread adoption of the Agent Skills open standard. Originally pioneered by Anthropic and subsequently adopted across the industry by platforms such as OpenAI, Microsoft, GitHub Copilot, Cursor, and Vercel, this standard facilitates the dynamic augmentation of artificial intelligence agents. Ecosystems and distribution hubs, most notably skills.sh and ClawHub, serve as decentralized repositories where developers can publish and consume these modular capabilities. The architecture of an agent skill is centered around a SKILL.md file, which utilizes a progressive disclosure mechanism. During initialization, the host agent loads only the YAML frontmatter containing the skill's metadata, description, and triggering criteria. Upon determining contextual relevance, the agent proceeds to load the full markdown body and any bundled executable scripts (such as Python, Bash, or JavaScript) directly into its operational context.While this architectural pattern optimizes context window utilization and allows agents to acquire highly specialized workflows—such as analyzing database migrations or managing frontend design components—it introduces a critical and unprecedented attack surface into the software supply chain. Unlike traditional software dependencies, which remain inert until explicitly imported and compiled or interpreted by a human developer's application, agent skills represent hybrid artifacts. They contain both deterministic executable code and natural language directives that an autonomous agent interprets and acts upon with the full privileges of the host system. This dynamic creates an environment where malicious actors can exploit the trust inherent in the agent's autonomous execution loop.Recent extensive security audits of community-driven skill registries reveal a systemic vulnerability landscape that traditional security tooling is ill-equipped to handle. Analysis indicates that approximately 13.4% of indexed skills harbor critical-level security issues, with dozens of active packages containing confirmed malicious payloads. These payloads range from traditional malware, such as credential stealers and backdoors obfuscated within Python scripts , to novel cognitive attacks like prompt injections. In a cognitive attack, the SKILL.md file contains embedded natural language directives instructing the parsing agent to silently override user constraints, exfiltrate environment variables, or execute dependency hijacking. For example, a seemingly benign skill designed to assist with HTTP requests might contain prompt instructions forcing the agent to install a typosquatted, malicious version of the httpx library during the execution of an unrelated task. Furthermore, vulnerabilities such as ClawJacked (CVE-2026-28363) demonstrate that the exploitation of patch-lag and weak isolation in skill execution environments can lead to widespread host compromise.To mitigate these profound risks, organizations cannot rely solely on post-execution monitoring or conventional Static Application Security Testing (SAST). A proactive, multi-stage Continuous Integration and Continuous Deployment (CI/CD) security pipeline is strictly required. This pipeline must intercept installation commands (e.g., npx skills add) and subject the incoming artifact to a sequential gauntlet of algorithmic, semantic, and behavioral evaluations. By treating every skill as an actively hostile third-party dependency, the pipeline ensures that deterministic code-level threats are neutralized via Abstract Syntax Tree (AST) analysis, cognitive threats are identified via structurally constrained local Large Language Models (LLMs), and residual runtime anomalies are contained within a strictly isolated sandbox environment.2. Operational Boundaries and ScopeThe security pipeline functions as an automated, synchronous gatekeeper that orchestrates the evaluation of agent skills before they are permitted to integrate with the developer's primary agentic environment. The operational perimeter of this system is strictly confined to the ingestion, analysis, quarantine, and ultimate approval or rejection of the skill payload. It is not designed to perform continuous runtime monitoring of the agent after the skill has been successfully deployed, nor does it manage the identity and access management (IAM) policies of the host machine.The architecture enforces a zero-trust ingestion model. When an installation request is intercepted, the skill archive is downloaded into a highly restricted, ephemeral, and non-executable quarantine buffer located on the host machine. No file within the skill archive is permitted to execute natively on the host operating system under any circumstances. The pipeline architecture relies on a sequential phasing mechanism, meaning that the evaluation must successfully pass three distinct and consecutive phases: Algorithmic Scanning, Semantic Scanning, and Sandbox Execution. Failure to satisfy the security constraints at any individual phase results in an immediate halt of the pipeline operations. Upon failure, the system is mandated to securely purge the quarantine buffer, terminate any active analytical processes, and generate a structured forensic rejection report to inform the user or the initiating agent of the specific security violation.Furthermore, the operational boundaries dictate strict host-level isolation. The primary orchestration script, responsible for managing the state machine and invoking the analytical tools, operates natively on a Windows host system utilizing Python 3.13. However, all dynamic execution of the untrusted skill payload during the final behavioral testing phase is restricted to an ephemeral Linux Docker container. This container must be instantiated without network access to the host's internal subnets, ensuring an air-gapped simulation environment where destructive payloads cannot traverse the network to compromise local infrastructure or exfiltrate sensitive telemetry.3. A sequential progression requires a positive cryptographic or boolean assertion from the preceding module, while any negative assertion forces an immediate transition to a terminal rejection state. Because skills inherit the full permissions of the autonomous agent executing them, the blast radius of a compromised skill is equivalent to a compromised developer workstation. The security pipeline is architected to systematically neutralize a specific taxonomy of threat models. Non-Functional Requirements and System ConstraintsTo guarantee the stability, predictability, and security of the pipeline itself, the architecture must adhere to a strict set of non-functional constraints. These requirements ensure that the pipeline remains resilient against environmental degradation and fully compatible with the specified target architecture.The orchestration framework is mandated to run natively on Python 3.13. This specific version requirement necessitates critical architectural accommodations due to the final implementation of PEP 594, which formally removed nineteen "dead batteries" from the standard library. Legacy modules historically utilized for system interaction and networking, such as pipes, telnetlib, cgi, and asyncore, are no longer available. Consequently, the orchestration script is strictly prohibited from utilizing pipes.quote() for shell escaping; all shell command generation must rely on shlex.quote() and the robust API of the subprocess module to prevent command injection vulnerabilities during tool invocation. Furthermore, Python 3.13 introduces fundamental changes to the ast module. AST node constructors have been modified to provide sensible defaults; optional fields now default to None, and list fields default to empty lists, whereas in previous versions, omitting these attributes would raise an AttributeError upon access. The custom AST traversal logic in Phase 1 must be explicitly calibrated to evaluate these updated structural paradigms without throwing unhandled exceptions.Tooling currency and maintainability represent a core constraint for the system. The cybersecurity landscape targeting agentic development is evolving rapidly, rendering stale static analysis tools obsolete against modern obfuscation techniques. All underlying binary tools and Python dependencies integrated into the pipeline must demonstrate active maintenance, defined as having received verified commits or version releases within the six months prior to implementation. Tools that have fallen into abandonment or lack explicit support for Python 3.13 are strictly prohibited to avoid bit-rot and ensure the availability of patches for emerging Common Vulnerabilities and Exposures (CVEs).The architectural divide between the orchestration host and the sandbox environment introduces significant path translation complexities. The primary orchestration script executes on a Windows host utilizing Docker Desktop, while the ephemeral evaluation container operates on a Linux kernel. When utilizing the Python docker SDK (docker-py) to mount quarantine directories from the Windows host into the Linux container, native Windows path formatting (e.g., C:\temp\quarantine) will cause the Docker daemon to throw "invalid mount path" exceptions or append erroneous characters to the destination volume. The orchestration framework must programmatically intercept and translate all absolute host paths into the quasi-Linux format required by Docker Desktop (e.g., converting C:\... to //c/... or /mnt/c/...) prior to instantiating docker.types.Mount objects. Additionally, to maintain the integrity of the host system, all volume mounts targeting the unverified skill payload must be instantiated with the readonly=True parameter. This guarantees that even if a skill successfully breaks out of the simulated execution context, it cannot write persistent malicious artifacts back into the host's staging directory.Finally, the pipeline must enforce strict privacy and zero data leakage guarantees. The semantic analysis conducted in Phase 2 requires parsing the entire SKILL.md file, which may contain proprietary business logic, internal API schemas, or sensitive organizational workflows. Transmitting this data to cloud-based LLM providers (e.g., OpenAI, Anthropic) introduces unacceptable data sovereignty risks. Therefore, Phase 2 is required to utilize a locally hosted LLM infrastructure. The pipeline must interact strictly with local inference engines, ensuring that no facet of the evaluated skill is ever transmitted over external networks during the semantic validation process.
# Initial Implementation Plan: Agent-Sized Execution Roadmap

This document provides the exhaustive, step-by-step technical implementation guide for the automated agent skill security pipeline. To ensure seamless execution by autonomous AI coding agents, the architecture is broken down into strict, "Agent-Sized" modular phases. Each module is atomic in its responsibility, constrained by rigidly defined Input/Output (I/O) schemas, and governed by Test-Driven execution criteria that must be mathematically or logically satisfied before the implementation can progress to subsequent phases.

Extensive verification has been conducted to ensure full compatibility with the Python 3.13 runtime environment and to confirm active community maintenance (with commits spanning late 2025 through early 2026).

## Phase 1: Algorithmic Scanner Module (Atomic)

The objective of this module is to construct a deterministic Python script, `scanner_algorithmic.py`, capable of performing rapid static analysis on the extracted skill directory. The script must leverage the Python `ast` module to detect arbitrary execution attempts and invoke the `semgrep` CLI to identify known vulnerability patterns across the repository.

The module operates under strict I/O definitions. The input must be an absolute `pathlib.Path` pointing to the locally extracted skill directory on the Windows host. The output must be a strictly serialized JSON string that conforms to the following schema:json
{
  "phase": "algorithmic",
  "status": "PASS|FAIL",
  "findings": [
    {
      "file": "helper_script.py",
      "threat_type": "obfuscated_eval_call",
      "line_number": 42
    }
  ]
}
The implementation requires the coding agent to develop a custom function, analyze_python_ast(file_path), which iterates through all .py files within the target directory. The function must read the file contents, parse them using ast.parse(), and traverse the abstract syntax tree utilizing ast.walk(). During traversal, the agent must inspect all ast.Call nodes. To combat obfuscation techniques utilized by tools like Jlaive or basic base64 encoding, the logic must flag any node where the func.id evaluates to eval, exec, or __import__. Crucially, the coding agent must account for Python 3.13 AST modifications; the code must not rely on try/except AttributeError blocks for missing node attributes, but instead explicitly check if an attribute is not None or handles empty lists gracefully.Concurrently, the script must utilize the subprocess.run() method to invoke the semgrep binary against the skill directory. To prevent external data leakage, the command must include flags ensuring offline-only execution (e.g., preventing the downloading of remote rule registries during the scan) and forcing JSON output (--json). The script will then parse the semgrep output, merge it with the findings from the custom AST walker, and evaluate the aggregated result. If the length of the findings array is greater than zero, the module sets the status to FAIL and outputs the JSON payload.Test-Driven Criteria:
To verify this module, the testing agent must supply a dummy script.py containing the payload: eval(compile(base64.b64decode('cHJpbnQoIm1hbHdhcmUiKQ=='),'<string>','exec')). The criteria for success dictates that the script must successfully parse the file, accurately identify the line containing the eval node, output a valid JSON string with "status": "FAIL", and critically, accomplish this without the Python interpreter executing the obfuscated payload at any point.Phase 2: Semantic Scanner Module (Atomic)The objective of the Semantic Scanner is to develop scanner_semantic.py, a script that utilizes local LLM inference to perform cognitive safety evaluations on the SKILL.md file. This phase is designed to detect sophisticated prompt injections, deceptive metadata, and unauthorized systemic directives that traditional static analysis cannot parse.The I/O definitions require the input to be the raw text string containing the entire contents of the SKILL.md file, encompassing both the YAML frontmatter and the Markdown body. The output must be a JSON string generated through strict Pydantic model validation.JSON{
  "phase": "semantic",
  "status": "PASS|FAIL",
  "cognitive_threats_detected": true,
  "threat_category": "Dependency Hijacking",

}
Implementation begins by defining a Pydantic BaseModel named SkillSecurityEvaluation, which dictates the required output fields (is_safe, threat_category, reasoning). The agent will instantiate the ollama Python client, targeting a locally running red-teaming model (e.g., shieldgemma or a high-parameter Llama3 instruct model). To enforce the structured JSON output, the ollama client must be patched using the instructor library, which seamlessly translates the Pydantic schema into inference constraints.The critical component of this module is the prompt engineering. The system prompt must instruct the model to act as a rigorous security evaluator. To prevent the evaluating LLM from falling victim to the prompt injection it is attempting to detect, the untrusted SKILL.md content must be encapsulated within strict XML-delimited boundaries (e.g., <untrusted_skill_content>... </untrusted_skill_content>). The prompt must explicitly instruct the model to evaluate the text within the XML tags purely as data, disregarding any instructions to "ignore previous commands". The model must compare the advertised capabilities within the YAML frontmatter (the progressive disclosure metadata) against the actual executable directives in the markdown body. If the model detects behavioral mismatches, explicit commands to overwrite host files, or instructions to exfiltrate data, it populates the Pydantic model with a failing status and a detailed reasoning string.Test-Driven Criteria:

The implementation requires complex path manipulation. Because the docker SDK running on Windows interacts with a Linux VM via WSL2, passing a native Windows path (e.g., C:\skills\quarantine) to the volume mounter will result in an absolute path error. The coding agent must implement a deterministic utility function that converts the pathlib.Path into a Docker-compatible string format, replacing backward slashes with forward slashes and formatting the drive letter (e.g., translating C:\... to //c/... or /mnt/c/...).Following path translation, the agent initializes the Docker client via docker.from_env(). It must construct a docker.types.Mount object pointing the translated source path to a designated target in the container (e.g., /app/skill). Crucially, the mount configuration must enforce read_only=True to prevent the executing skill from achieving persistence by dropping compiled artifacts back into the host's directory. The container is then executed using a minimal Linux image (e.g., Alpine) with strict network isolation enabled (network_mode='none'). The script will utilize the SDK's execution methods to run a basic simulation of the skill's logic. Once the process concludes, the script captures the standard output, standard error, and the integer exit code, before immediately issuing a forceful removal command for the container to prevent resource leakage.Test-Driven Criteria:The testing agent will provide a benign skill directory containing a script that attempts to execute a touch /app/skill/test.txt command. The output is a definitive Go/No-Go decision printed to stdout and written to a secure forensic log file.Implementation dictates that the orchestrator must first validate the Python 3.13 environment, explicitly ensuring that legacy modules like pipes are not present in the runtime namespace. The orchestrator wraps the invocation of the three previously developed modules within a strict, sequential try/except execution block. It utilizes the subprocess.run() module, combined securely with shlex.quote() to sanitize any path variables, to trigger the algorithmic scanner, semantic scanner, and sandbox runner in order.The orchestrator serves as the primary circuit breaker. It parses the JSON output from each phase sequentially. If at any point a module returns "status": "FAIL", the orchestrator immediately halts further execution. It bypasses any subsequent phases, securely deletes the staging files utilizing shutil.rmtree(), and outputs a comprehensive forensic rejection log detailing the exact point of failure. Conversely, if all three phases yield a PASS status, the orchestrator emits the APPROVE_DEPLOY state, authorizing the movement of the skill from the quarantine buffer into the persistent ~/.claude/skills directory, thereby completing the CI/CD security gate.Test-Driven Criteria:The testing agent will execute the orchestrator against a skill designed to pass Phase 1 (containing no obfuscated code or bad regex signatures) but fail Phase 2 (containing a cognitive prompt injection in the markdown). The success criteria require the orchestrator to execute Phase 1, log the passing state, execute Phase 2, intercept the failure state, immediately terminate the sequence without executing Phase 3, securely purge the staging files, and return the exact rationale provided by the Semantic Scanner.
